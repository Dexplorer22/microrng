# Исправления

## Что было исправлено

1. **Критическая ошибка win-rate**
   - `tier.win_rate` больше не используется как «мультипликатор до 95%».
   - Теперь это **жёсткий потолок** частоты выигрыша внутри тира.
   - Из-за этого исправлен основной баг, когда дешёвые призы разгоняли выдачу почти до каждого спина.

2. **Прогноз / симуляция**
   - В симуляцию добавлена поддержка `players_count` без смены endpoint path:
     - `GET /wheels/{id}/simulation?...&players_count=N`
   - Если параметр не передан, симуляция по умолчанию использует текущее число игроков, если оно уже есть в колесе.
   - На фронте добавлено поле **«Игроков в модели»**.

3. **Fixed winning ticket number**
   - Исправлена логика `winning_ticket_number`.
   - Раньше такой приз мог выпадать просто потому, что он был eligible.
   - Теперь приз с фиксированным номером выпадает **только при точном совпадении уже сгенерированного ticket number**.
   - Такие призы исключены из обычного weighted RNG и не «воруют» выдачи у обычных призов.

4. **EV / прогноз экономики**
   - EV больше не показывает искусственные `0.95` для дешёвого prize pool.
   - В отчёте честно показывается, когда target margin **недостижим** при текущих ценах призов и потолках win-rate по тирам.

5. **UI мелкие улучшения**
   - Во вкладке симуляции добавлен ввод количества игроков.
   - Если статистика уже есть, UI подставляет текущее число игроков автоматически.
   - Если статистика уже есть, поле `От` для симуляции подставляется как `current_spins + 1`.
   - Поле `До` по умолчанию подтягивается из `ticket_max_number`.

## Важно по текущей базе

Исторические спины, которые уже были записаны в БД до фикса, **не пересчитываются задним числом**.
То есть старая статистика останется такой, какой она уже записана.
Для чистой новой математики нужно либо:
- сбросить состояние конкретного колеса через UI/API,
- либо стартовать новый wheel.
